One bash script · cron every 5 min · restarts only the unhealthy agent

Section 04

Deployment Rules

Non-negotiable constraints applied on every deploy to every agent

Rule · 01

One clone per repo

Every GitHub repo has exactly one working directory on the VM. Agents sharing a repo share one directory and differ only by process name. Duplicate clones are permanently forbidden.

Rule · 02

Build before restart

git pull → build passes → pm2 restart --update-env. If the build fails, the running process stays up. Never restart a broken build into production.

Rule · 03

Secrets isolation

Each agent has its own .env file. No secrets in source control. Database wallet files deployed separately from code. TypeScript strict mode — no any types, no hardcoded credentials.

Rule · 04

PM2 persistence

pm2 startup + pm2 save required on every new agent. Processes survive server reboots. max_restarts and autorestart: true in every ecosystem config.

Rule · 05

No silent failures

All agents catch and log uncaught exceptions before exit. PM2/systemd sees the cause before restart. Health script provides 5-minute SLA on failure detection.

Rule · 06

Verify after every deploy

After deploy: check HTTP health endpoint returns 200 · check PM2 logs show "Connected" · test one Telegram command end-to-end. No deploy is done until all three pass.

Section 05

Postmortem Format

Example: HubSpot Duplicate Loop — May 10, 2026

🔁

HubSpot Duplicate Posting Loop

Same tweet posted twice, ~6 minutes apart, on every automation cycle

Symptom
HubSpot milestone items kept reappearing as pending on every cycle. Same tweet posted twice within 6 minutes.
Root causes
Three compounding bugs: (1) field name mismatch — pipeline wrote posted: true, GET filter checked posted_x — items always passed the filter. (2) Timestamp field mismatch — mark endpoint matched on timestamp, old items only had received_at — mark never stuck. (3) Five backlog items had neither field set — they would serve as pending forever.
Fix applied
Updated GET filter to exclude both fields. Rewrote mark endpoint with 3-tier matching: timestamp → received_at → title fallback. Backfilled 5 stale records manually.
Verified by
Live API call confirmed: {"pending": [], "total": 0}. All items show both fields set. Monitored for two full cycles — no duplicates.
Resolution
~2 hours from symptom to confirmed fix. Documented same day in infrastructure SOP.
📝
SOP update rule: Every postmortem adds a new section to this document with root cause, fix, and verification steps — so the same issue takes zero hours next time.
