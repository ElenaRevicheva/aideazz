Deployment Methodology

Rule #1

One clone per repo

Every GitHub repo has exactly one working directory on the VM. Agents sharing a repo share one directory and differ only by process name. Duplicate clones are forbidden.

Rule #2

Deploy sequence

git pull → npm run build (or equivalent) → pm2 restart --update-env. Build must pass before restart. TypeScript strict mode always.

Rule #3

Environment isolation

Each agent has its own .env. Secrets never in source control. Database wallet files deployed separately from code. Credentials rotated on compromise, not on schedule.

Rule #4

PM2 persistence

pm2 startup + pm2 save on every new agent. Processes survive server reboots. max_restarts and autorestart: true in ecosystem config.

Rule #5

No silent failures

All agents log uncaught exceptions before exit. PM2/systemd sees the cause before restart. Health script provides 5-min SLA on detection.

Rule #6

Verify after deploy

After every deploy: check HTTP health endpoint returns 200, check PM2 logs show "Connected" state, test one Telegram command to confirm end-to-end path.

Postmortem Format — Example: HubSpot Duplicate Loop (May 10, 2026)

Symptom Same tweet posted twice, ~6 minutes apart. HubSpot milestone items kept reappearing as pending on every automation cycle.
Root causes Three compounding bugs: (1) field name mismatch — pipeline wrote posted: true, filter checked posted_x — items always passed. (2) timestamp field mismatch — mark endpoint matched on timestamp, old items only had received_at — mark never stuck. (3) Five backlog items had neither field set.
Fix Updated GET filter to exclude both fields. Rewrote mark endpoint with 3-tier matching: timestamp → received_at → title fallback. Backfilled 5 stale records manually.
Verification Live API call confirmed: {"pending": [], "total": 0}. All 4 HubSpot items show both fields set. Monitored for two cycles — no duplicates.
Time to resolution ~2 hours from symptom to confirmed fix. Documented same day.
SOP update triggered: Every postmortem adds a new section to this document with root cause, fix, and verification steps — so the next person (or future self) doesn't spend 2 hours on the same issue.
